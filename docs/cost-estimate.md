# 运行成本估算

更新时间：2026-07-01

这份文档按当前仓库代码估算运行成本。结论先说清楚：当前方案不需要单独租云服务器，前端是微信小程序，后端使用微信云开发/CloudBase 的云函数、云数据库和可选云存储；真正需要按量关注的是云函数调用、数据库读写、图片/文件存储与下载流量、日志，以及微信支付商户费率。

## 当前结论

- 服务器：不需要单独购买 CVM 云服务器，除非以后要做独立后台、复杂报表、ERP 对接或非微信生态接口。
- 数据库：订单、商品、支付、退款等数据保存在云开发数据库集合里，当前集合见 [deployment.md](./deployment.md)。
- 支付：当前代码走微信官方支付链路，后端调用 `cloud.cloudPay.unifiedOrder`，小程序前端调用 `wx.requestPayment` 拉起微信支付。
- 支付 API 调用费：当前资料不能证明“每调用一次支付 API 单独收费”。实际成本应按微信支付商户平台签约费率从成功交易金额中扣取，费率以商户后台/签约协议为准。
- 图片：当前代码支持商品 `images` 字段和后台录入图片地址，但商品列表页面暂未渲染 `<image>`，所以现版本用户打开店铺不会产生商品图片下载流量。后续展示图片后，图片成本按存储量、下载流量和请求次数计算。
- 小程序认证/主体费用：属于上线资质成本，不是代码运行成本。是否需要认证费、金额和周期以微信公众平台后台当前提示为准。

## 官方依据

- 微信小程序 `wx.requestPayment` 官方文档说明：调用前需在小程序微信公众平台申请接入微信支付；如果使用云开发，可以通过云开发微信支付统一下单接口获取前端支付所需参数。链接：https://developers.weixin.qq.com/miniprogram/dev/api/payment/wx.requestPayment.html
- 微信支付小程序支付开发指引说明：商户先调用 JSAPI/小程序下单获取 `prepay_id`，再通过小程序 `requestPayment` 唤起微信支付；支付成功后微信支付会发送回调通知；支付成功 1 年内可申请退款。链接：https://pay.weixin.qq.com/doc/v3/merchant/4012791911
- 腾讯云 CloudBase 价格页入口：https://cloud.tencent.com/product/tcb/pricing 。2026-07-01 从命令行抓取该页面时返回了安全校验页，本文不硬编码 CloudBase 单价，最终单价需要在腾讯云/微信云开发控制台确认。

## 一次用户下单的大致资源量

假设用户打开店铺、选择商品、提交订单并支付成功：

- 店铺浏览：2 次 `api` 云函数调用。
- 创建订单：1 次 `api` 云函数调用，读商品，写订单。
- 创建支付：1 次 `api` 云函数调用，读订单，写支付记录，调用微信支付统一下单。
- 支付成功回调：1 次 `payNotify` 云函数调用，更新订单和支付记录。
- 合计：约 5 次云函数调用，若不计算商品列表分页，一般是数次数据库读取和 3 到 5 次数据库写入/更新。

如果用户取消支付：

- 已发生店铺浏览、创建订单、创建支付和 `wx.requestPayment` 拉起。
- 不会有支付成功回调。
- 订单停留在 `pending_payment`，后续需要通过订单清理/关单能力处理，这部分当前代码还没有定时任务。

## 图片成本怎么算

当前页面没有展示商品图片，所以商品图片下载成本暂时约等于 0。

以后如果在商品列表展示图片，按下面公式估算：

```text
月图片下载流量 GB = 月店铺打开次数 * 每次实际加载图片数 * 单张图片大小 KB / 1024 / 1024
月图片下载成本 = 月图片下载流量 GB * 云存储/CDN 下载流量单价
月图片存储成本 = 图片总大小 GB * 云存储单价
```

例子：

- 每月 600 次店铺打开。
- 每次加载 10 张商品图。
- 每张压缩后 100 KB。
- 月下载流量约 `600 * 10 * 100 / 1024 / 1024 = 0.57 GB`。

如果每张图是 300 KB，则同样访问量约 1.72 GB。图片越大，成本和用户加载速度都会明显变差。建议商品列表图控制在 100 KB 到 200 KB 左右，优先使用压缩后的 WebP/JPEG 缩略图，不要直接展示原图。

## 支付成本怎么算

当前代码的支付链路是：

1. 后端 `createPayment` 调用 `cloud.cloudPay.unifiedOrder`。
2. 前端拿到支付参数后调用 `wx.requestPayment`。
3. 用户支付成功后，微信支付回调 `payNotify`。

成本拆分：

- 支付下单/拉起本身：按当前官方文档和可访问资料，不能确认存在“每次 API 调用固定扣费”。估算时按云函数调用和微信支付商户费率处理。
- 成功交易手续费：`月支付手续费 = 月成功交易金额 * 商户签约费率`。
- 退款：当前代码支持整单退款，调用 `cloud.cloudPay.refund` 并接收 `refundNotify`。退款是否退还手续费、退还规则、结算规则以微信支付商户协议和商户后台为准。

示例只用于理解：

```text
日订单数 = 10
客单价 = 50 元
月成功交易金额 = 10 * 50 * 30 = 15000 元
若商户签约费率是 0.6%，月支付手续费 = 15000 * 0.006 = 90 元
```

这里的 0.6% 只是示例参数，不代表当前商户真实费率。真实费率必须看微信支付商户平台的签约结果。

## 云开发成本怎么算

最终费用取决于云开发环境套餐或按量计费规则。当前代码会用到：

- 云函数调用次数：`api`、`payNotify`、`refundNotify`、`platformNotify`。
- 云函数资源用量：执行时间、内存规格、并发和日志量。
- 云数据库：集合存储量、读次数、写次数。
- 云存储：如果商品图放在云存储，会产生存储量、下载流量和请求次数。
- 日志：云函数日志保留和检索可能产生费用，取决于控制台配置。

建议上线前在腾讯云/微信云开发控制台确认这些单价：

- 云函数调用次数单价和免费额度。
- 云函数 GBs/资源使用量单价。
- 云数据库读写次数、容量、备份相关价格。
- 云存储容量、下载流量、请求次数价格。
- 日志服务是否单独计费。

## 小规模月度估算模板

可以先用下面参数做预算：

```text
月店铺打开次数 = A
月成功订单数 = B
客单价 = C 元
商户支付费率 = R
平均每单商品种类数 = I
每次打开加载图片数 = P
单张图片大小 = S KB
```

估算：

```text
月云函数调用次数
  ~= A * 2
   + B * 3
   + 商家后台操作次数
   + 退款次数 * 2

月成功交易金额
  = B * C

月支付手续费
  = B * C * R

月图片下载流量 GB
  = A * P * S / 1024 / 1024
```

举例，微信群内小范围使用：

```text
A = 600 次店铺打开/月
B = 300 单/月
C = 50 元
R = 0.6% 示例费率
P = 0，当前页面不展示商品图
S = 0

月云函数调用次数粗估：600 * 2 + 300 * 3 = 2100 次，加后台操作和回调冗余后仍是小规模量级
月成功交易金额：300 * 50 = 15000 元
月支付手续费示例：15000 * 0.006 = 90 元
月商品图片下载流量：0 GB
```

如果后续展示商品图，假设 `P = 10`、`S = 100 KB`：

```text
月图片下载流量：600 * 10 * 100 / 1024 / 1024 = 0.57 GB
```

## 成本控制建议

- 上线初期继续用云开发，不租云服务器。
- 商品图上线前先做缩略图和压缩，列表图控制在 100 KB 到 200 KB。
- 店铺首页可以做短时间缓存，避免用户反复进群链接时重复拉取商品列表。
- 云函数日志保留时间不要设置过长，生产环境避免大量 `console.log`。
- 在腾讯云/微信云开发控制台开启预算提醒和用量告警。
- 定期导出或归档历史订单，避免订单集合无限增长影响查询和存储。
- 后续补定时关单任务，清理长期未支付订单，避免后台订单列表和数据库持续膨胀。

## 需要上线前确认的项目

- 微信公众平台：小程序主体认证是否需要费用、费用金额和有效期。
- 微信支付商户平台：该商户类目的真实交易费率、结算周期、退款手续费规则。
- 云开发控制台：当前环境套餐、免费额度、超额单价和日志计费规则。
- 图片存储方案：商品图使用云存储、腾讯云 COS/CDN，还是外部 HTTPS 图床；不同方案的下载流量单价不同。
